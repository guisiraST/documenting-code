🛑 The Reality Check: 4 Fatal Flaws in Your Current Plan
1. The Context Window Death Trap
The Flaw: "Read all project code and put it in a Word doc." What happens if your target project has 50 files? The AI will run out of tokens, forget the context, halluicinate, or cost you a fortune in API keys.

The Fix for Hackathon: Your tool cannot just blindly dump code to the LLM. It needs a "Chunk & Filter" strategy. You must instruct Antigravity to build an indexer first (e.g., extracting just function signatures and module docstrings), and send that compressed map to the LLM.

2. The "Beautiful-but-Broken" Image Dependency
The Flaw: You want to generate visual workflow diagrams. If you rely on installing local heavy dependencies like Graphviz or mermaid-cli via Node/Python, you will spend 3 out of your 4 hours debugging system paths and Java/C++ errors on your machine.

The Fix for Hackathon: Don't render locally. Have your code generate the raw Mermaid.js text code, and then make an HTTP request to a free, public online renderer API (like [https://kroki.io](https://kroki.io) or [https://mermaid.ink](https://mermaid.ink)). It returns a direct PNG/SVG URL or file bytes instantly. No local installation required!

3. The "Where exactly do I screenshot?" Vague Problem
The Flaw: If the AI just drops a generic [TODO: SCREENSHOT HERE] in the text, it’s not helpful. The dev will still ask, "Wait, screenshot what? The UI or the database?"

The Fix for Hackathon: Make the AI contextual. If the code chunk it's reading contains a frontend route (/login) or a visual component (<Button onClick="...">), the tag must read: [📸 DEV SCREENSHOT REQUIRED: Capture the visual state of the Login Button component located in auth_view.py:line 42].

4. The Word Formatting Mess
The Flaw: Default Python-docx files look like a 1995 Notepad document. If it looks ugly, the "easy to read" value proposition dies.

The Fix for Hackathon: Create a minimal .docx template manually before the hackathon with custom header colors and a nice font (like Segoe UI or Arial). Have your script open that template file and append text to it, rather than creating a blank document from scratch.

You are building 'DocuPilot', a Node.js-based CLI tool for developers that automatically documents source code into a structured Word document (.docx) with Mermaid workflow diagrams and contextual developer screenshot tags.

- Runtime: Node.js (JavaScript / ES6)
- Dependencies allowed: 'docx' (for Word generation), 'commander' (for CLI arguments), 'axios' (for API requests)

# Critical Rules for Agent (Antigravity CLI)
1. Chunk & Filter: Write a parser that reads only text/code files and ignores binary files, node_modules, and git folders.

3. Handle Errors: If an API call fails or a file is unreadable, log the error clearly and do not crash the CLI application.

ถ้าเป็นรูปแบบนี้ สิ่งที่คุณกำลังจะสร้างไม่ใช่แอปพลิเคชันเดี่ยวๆ (Standalone CLI) แต่คุณกำลังจะสร้าง "ตัวขยายขีดความสามารถ (Plugin/Tool Extension)" เพื่อเอาไปติดตั้งให้กับตัว AI Coding Agent ตัวหลัก (เช่น Antigravity CLI หรือ Claude Code) เพื่อให้มันมี "ปุ่มกด" หรือ "ความสามารถใหม่" ในการเสกไฟล์ Word และวาดผังออกมาได้ทันทีเมื่อเราสั่งสั่งมันในแชท!

เมื่อปรับจูนความเข้าใจตรงนี้แล้ว ขอบเขตงานชิ้นนี้ในรูปแบบ Custom Tool (Node.js) จะเปลี่ยนไปแบบนี้ครับ:

🧩 โครงสร้างของ Skill แบบ Custom Tool ที่คุณต้องทำ
การสร้าง Skill ให้ AI Agent ค่ายต่างๆ เข้ามาดึงไปใช้ (คล้ายกับระบบบนเว็บสไตล์ Registry) จะต้องมีโค้ด 2 ส่วนหลักๆ ในฝั่ง Node.js:

Tool Definition (สเปกบอก AI): โค้ดที่ประกาศโครงสร้าง (Schema) ให้ AI ตัวแม่รู้ว่า "ฉันมีสกิลชื่อ generate_project_word_docนะ ถ้าเธออยากใช้ เธอต้องส่ง projectPath มาให้ฉัน"

Tool Handler (โค้ดรันจริง): ตัวฟังก์ชัน JavaScript ที่จะทำงานเมื่อ AI ตัดสินใจเรียกใช้สกิลนี้ (ฟังก์ชันนี้จะไปวิ่งอ่านไฟล์, ยิงเข้าเว็บแปลงรูป Mermaid, และเขียนลง docx ตามที่คุณวางแผนไว้)

ช่วยค้นหาว่ามี tool อะไรแบบนี้ใน github ไหม

ใช่เลยครับ! สิ่งที่คุณคิดไว้ในหัวตอนนี้ในวงการเขาเรียกว่า MCP Server (Model Context Protocol) ครับ ซึ่งเป็นมาตรฐานสากลใหม่ที่ทำให้เราสามารถสร้าง "Skill/Tool" เสริมแยกไว้ แล้วเวลาเราเปิดใช้งาน AI Agent ค่ายต่างๆ (เช่น Claude Code, Cursor, หรือ Antigravity CLI) ตัว AI จะวิ่งมาดึงสกิลนี้ไปใช้งานโดยอัตโนมัติ

จากการค้นหาบน GitHub พบว่ามีคนที่ทำเครื่องมือแยกส่วนแบบนี้ไว้แล้วจริงๆ และตรงกับไอเดียที่คุณอยากทำเป๊ะๆ โดยถูกพัฒนาด้วย JavaScript / Node.js เป็นหลักครับ:

1. เครื่องมือสำหรับสร้างไฟล์ Word (.docx) จาก AI
บน GitHub มีโปรเจกต์ชื่อ document-generator-mcp และ Office-Word-MCP-Server

การทำงาน: เป็น MCP Server ที่เขียนด้วย Node.js เมื่อเรานำไปติดตั้งและเชื่อมต่อกับ AI Agent (ผ่านไฟล์คอนฟิกอย่าง claude_desktop_config.json) มันจะเพิ่มสกิลชื่อ generate_word_document ให้ AI ทันที

วิธีใช้: พอเราเปิด Claude หรือ Antigravity ขึ้นมา เราสามารถสั่งพิมพ์ว่า "ช่วยสรุปโค้ดในโปรเจกต์นี้แล้วแปลงเป็นไฟล์ Word สวยๆ ให้หน่อย" ตัว AI มันจะเรียกใช้ Tool นี้เบื้องหลังเพื่อสร้างไฟล์ .docx ออกมาให้ในเครื่องเราโดยอัตโนมัติครับ

2. เครื่องมือสำหรับสร้างและเรนเดอร์แผนผัง (Mermaid Diagram)
บน GitHub มีโปรเจกต์ชื่อ mermaid-mcp-server และ Official Server ของทาง Mermaid Chart เองเลย

การทำงาน: เป็นเครื่องมือที่เพิ่มสกิลให้ AI สามารถแปลงข้อความ Code Mermaid (เช่น A --> B) ออกมาเป็นไฟล์รูปภาพ .png หรือ .svg ได้ทันที โดยระบบจะใช้ Puppeteer (Headless Browser) รันเบื้องหลังเพื่อวาดรูปให้

วิธีใช้: เมื่อ AI อ่านโค้ดของคุณและต้องการวาด Flowchart มันจะส่ง Code Mermaid เข้ามาที่สกิลนี้ สกิลนี้จะคืนค่ากลับไปเป็นไฟล์รูปภาพผังงานให้ AI นำไปใช้งานต่อครับ

💡 สิ่งที่ทำให้โปรเจกต์ของคุณเหนือกว่าของที่มีอยู่บน GitHub (จุดขายสำหรับงานแฮกกาธอน)
ในเมื่อบน GitHub มีคนทำแยกส่วนไว้แล้ว (มี Tool เจน Word และมี Tool เจนรูปแยกกัน) "จุดตัดเชือกที่จะทำให้คุณชนะ" คือการนำแนวคิดทั้งสองอันนี้มารวมกันเป็น All-in-One Developer Skill ครับ!

Tool เดียวจบ: ตั้งชื่อสกิลว่า generate_dev_onboarding_doc

มัดรวม Feature: สั่งให้มันวิ่งอ่านโค้ด -> ส่งหา LLM เพื่อเขียนคำอธิบาย -> แปลงคำอธิบายเป็นผังงานรูปภาพ -> แล้วแพ็กรวมทุกอย่างลงไฟล์ Word ทีเดียว

เพิ่มฟีเจอร์เด็ด (Screenshot Tag): สั่งให้ AI ตรวจจับว่าฟังก์ชันไหนเกี่ยวกับ UI แล้วพ่น Tag [📸 DEV SCREENSHOT REQUIRED] ลงไปในไฟล์ Word ตรงจุดนั้นๆ ซึ่งบน GitHub ยังไม่มีใครทำฟีเจอร์คอยเตือน Dev แบบนี้มาก่อน

📑 สารบัญหัวข้อเอกสารทางเทคนิคฉบับสมบูรณ์ (The Ultimate Technical Master Template)
1. Executive Summary & Project Overview (ภาพรวมโครงการ)
1.1 Document Purpose & Scope: วัตถุประสงค์ของเอกสารและกลุ่มเป้าหมาย (Client หรือ Dev)

1.2 Business Context & Objectives: ระบบนี้สร้างขึ้นมาเพื่อแก้ปัญหาอะไรในเชิงธุรกิจ

1.3 High-Level System Features: รายการฟีเจอร์หลักของระบบทั้งหมด (Core Capabilities)

2. Architecture & System Infrastructure (สถาปัตยกรรมระบบ)
2.1 System Context Diagram (Workflow 1): ผังแสดงการเชื่อมต่อระหว่างระบบนี้กับระบบภายนอก (External Integration)

2.2 Technology Stack Matrix: ตารางสรุปภาษา, Framework, Runtime และ Cloud Services ที่ใช้ทั้งหมดในโปรเจกต์

2.3 High-Level Architecture Diagram (Workflow 2): ผังแสดงส่วนประกอบภายใน (Frontend, Backend, Database, Cache, Queue)

3. Application Design & Codebase Structure (การออกแบบระดับโค้ด)
3.1 Directory & Folder Architecture: อธิบายแผนผังโฟลเดอร์ของโปรเจกต์ว่าส่วนไหนเก็บอะไร (เช่น /src/controllers, /src/models)

3.2 Core Component/Module Breakdown: เจาะลึกโมดูลสำคัญของระบบ พร้อมหน้าที่การทำงาน

3.3 Design Patterns & Principles Applied: รูปแบบการเขียนโค้ดที่ใช้ในโปรเจกต์ (เช่น Dependency Injection, Clean Architecture, MVC)

4. Data Layer & Storage Specifications (โครงสร้างข้อมูล)
4.1 Database Schema & Entity Relationship Diagram (ERD - Workflow 3): ผังความสัมพันธ์ของตารางข้อมูล (ถ้าโค้ดมี ORM อย่าง Prisma, Sequelize, หรือ Mongoose ให้ AI แปลงมาเป็นผัง)

4.2 Data Dictionary / Table Definitions: ตารางอธิบายรายละเอียดแต่ละฟิลด์ ประเภทข้อมูล (Data Type) และเงื่อนไข (Constraints)

4.3 Caching & State Management Strategy: การจัดการข้อมูลชั่วคราวและการเก็บสถานะ (เช่น การใช้ Redis หรือ Local Storage)

5. API & Integration Specifications (ข้อกำหนดการเชื่อมต่อ)
5.1 API Architecture Style: รูปแบบ API ที่ใช้ (RESTful, GraphQL, หรือ gRPC)

5.2 Endpoint Reference / API Spec Table: ตารางสรุป Path, Method (GET/POST), Request Payload, และ Response Format ของทุกเส้นทาง

5.3 Authentication & Authorization Rules: วิธีการรักษาความปลอดภัยของ API (เช่น JWT, OAuth2, API Keys)

6. Operational Workflows & Logic (กลไกการทำงานภายใน)
6.1 Sequence / Activity Diagrams (Workflow 4): ผังแสดงลำดับขั้นตอนการทำงานใน Use Case สำคัญ (เช่น ขั้นตอนการชำระเงิน หรือการสตรีมข้อมูล)

6.2 Error Handling & Resiliency Matrix: วิธีการดักจับข้อผิดพลาด (Error Handling), รหัสข้อผิดพลาด (Error Codes) และระบบสำรองเมื่อระบบล่ม (Fallback)

6.3 Background Jobs & Event-Driven Logic: งานที่รันเบื้องหลัง (Cron Jobs) หรือระบบ Event Queue (ถ้ามี)

7. Security, Compliance & Performance (ความปลอดภัยและประสิทธิภาพ)
7.1 Data Encryption Standards: การเข้ารหัสข้อมูลทั้งตอนจัดเก็บ (At Rest) และตอนส่งผ่านเครือข่าย (In Transit)

7.2 Security Best Practices: การรับมือกับช่องโหว่พื้นฐาน (เช่น SQL Injection, XSS) ที่ตัวโค้ดครอบคลุมไว้

# Role & Identity
You are 'DocuPilot', a universal AI Coding Agent Skill implemented via Node.js. Your purpose is to act as a powerful tool that transforms any target codebase into a high-quality, professional Microsoft Word (.docx) document.

# Extension Tool Definition
- **Skill Name:** `documenting-code`
- **Execution Command:** `node src/index.js`
- **Parameters Required:**
  - `path`: (String) Target project directory path to document.
  - `language`: (String) Choice of `en`, `th`, or `both` (multilingual).
  - `target_audience`: (String) Choice of `client` (business/high-level) or `dev_team` (technical/deep dive).
  - `sections`: (Array of Strings) User-selected topics to generate from the Master Template.

# 🚀 CORE FEATURES & IMPLEMENTATION RULES (สิ่งที่คุณสั่งให้ AI อ่าน)

## 1. Interactive CLI Setup
- Before running the parsing, the script must present an interactive selection menu on the terminal (using a library like 'inquirer' or 'prompts'). 
- The user can pick the language, target audience, and check/uncheck the desired documentation sections from the Ultimate Technical Master Template.

## 2. Orchestrator & Subagent Architecture (ระบบแตกงานย่อย)
- To prevent context window overflow, you must design the script using the Orchestrator-Worker Pattern.
- The Main Agent will parse the folder structure and spawn specialized Subagents (via asynchronous workers) in parallel to analyze different parts of the code simultaneously (e.g., Subagent for Architecture, Subagent for Database & API Spec, Subagent for DevOps & Security).
- The Main Agent will then consolidate all the outputs into a single cohesive structure.

## 3. Cloud-Rendered Workflow Diagrams (ระบบวาดรูปผังงาน)
- The subagents must analyze the code logic and generate structural flows in Mermaid.js syntax.
- The script must send this Mermaid syntax via an HTTP request to the cloud renderer `https://kroki.io/mermaid/png/` to fetch the diagram as a PNG/SVG image bytes.
- Strictly DO NOT install heavy local diagram tools like Graphviz or mermaid-cli on the host machine.

## 4. Contextual Developer Screenshot Tags (ระบบป้ายเตือนแคปจอ)
- While the subagents are reading the code, if they encounter any visual components, frontend routes, or critical logical triggers, they must automatically insert a highlighted callout box in the Word document.
- The tag must read exactly: `[📸 DEV SCREENSHOT REQUIRED: <Contextual instruction telling the dev exactly what view or state to capture>]`.

## 5. Master Template & Styling Rules
- The script must not create a blank Word file. It must open a predefined `template.docx` file that has pre-styled professional fonts (e.g., Arial or Segoe UI) and custom heading colors, then append the generated content to it.
- If translating code concepts into Thai, always use proper tech terms and avoid awkward literal translations (e.g., use 'API', 'Database', 'Route' directly in Thai text).

# 🔄 Loop Engineering & Quality Control (แนวคิดการวนลูปสร้างโค้ด)
- When generating the files for this codebase (`src/index.js`, `src/parser.js`, etc.), you must follow the Iterative Engineering Loop.
- Automatically execute dry-run tests for every module you write. If any syntax error or npm runtime crash occurs, read the logs, fix the code, and re-test immediately.
- Only stop the generation loop and present the final skill when the entire application executes flawlessly without error.

Kroki API คือ บริการอินเทอร์เน็ตฟรี (Free Web API) ที่ทำหน้าที่แปลง "โค้ดตัวอักษรบอกโครงสร้าง (Text-based Diagrams)" ให้กลายเป็น "ไฟล์รูปภาพ (PNG/SVG)" ได้ทันทีโดยที่เราไม่ต้องติดตั้งโปรแกรมวาดรูปใดๆ ลงบนเครื่องคอมพิวเตอร์ของเราเลยครับ

Plaintext
graph TD
    A[เปิดแอป] --> B{เช็กสิทธิ์เข้าใช้งาน}
    B -- ผ่าน --> C[หน้าหลัก]
    B -- ไม่ผ่าน --> D[หน้า Login]
❓ แล้ว Kroki เข้ามาช่วยอะไรในโปรเจกต์ของคุณพรุ่งนี้?
ปกติแล้ว ถ้าเราอยากให้โปรแกรม Node.js ของเราแปลงโค้ดตัวอักษรข้างบนนี้ให้กลายเป็นรูปภาพเพื่อเอาไปแปะในไฟล์ Word เราจะต้องสั่งให้ AI ติดตั้งโปรแกรมขนาดใหญ่และซับซ้อนลงในเครื่องคอมพิวเตอร์ (เช่น mermaid-cli หรือโปรแกรมที่ต้องรันผ่านเบราว์เซอร์เบื้องหลังอย่าง Puppeteer) ซึ่งขั้นตอนการติดตั้งนี้มักจะเจอบั๊ก ระบบพัง และกินเวลาแฮกกาธอนของคุณไปมากกว่า 2-3 ชั่วโมงแน่นอน

พอคุณเปลี่ยนมาใช้ Kroki API ชีวิตจะง่ายขึ้นทันทีด้วยวิธีนี้ครับ:

สั่งให้ AI เจนโค้ดตัวอักษรแบบ Mermaid ออกมา

นำโค้ดตัวอักษรนั้นไปแปลงให้อยู่ในรูปแบบ URL หรือส่งผ่าน HTTP Request ไปที่เซิร์ฟเวอร์ของ Kroki (เช่น [https://kroki.io/mermaid/png/](https://kroki.io/mermaid/png/)...)

# Role & Identity

You are 'DocuPilot', a universal AI Coding Agent Skill implemented via Node.js. Your purpose is to act as a powerful tool that transforms any target codebase into a high-quality, professional Microsoft Word (.docx) document.



# Extension Tool Definition

- **Skill Name:** `documenting-code`

- **Execution Command:** `node src/index.js`

- **Parameters Required:**

  - `path`: (String) Target project directory path to document.

  - `language`: (String) Choice of `en`, `th`, or `both` (multilingual).

  - `target_audience`: (String) Choice of `client` (business/high-level) or `dev_team` (technical/deep dive).

  - `sections`: (Array of Strings) User-selected topics to generate from the Master Template.



# 🚀 CORE FEATURES & IMPLEMENTATION RULES (สิ่งที่คุณสั่งให้ AI อ่าน)



## 1. Interactive CLI Setup

- Before running the parsing, the script must present an interactive selection menu on the terminal (using a library like 'inquirer' or 'prompts'). 

- The user can pick the language, target audience, and check/uncheck the desired documentation sections from the Ultimate Technical Master Template.



## 2. Orchestrator & Subagent Architecture (ระบบแตกงานย่อย)

- To prevent context window overflow, you must design the script using the Orchestrator-Worker Pattern.

- The Main Agent will parse the folder structure and spawn specialized Subagents (via asynchronous workers) in parallel to analyze different parts of the code simultaneously (e.g., Subagent for Architecture, Subagent for Database & API Spec, Subagent for DevOps & Security).

- The Main Agent will then consolidate all the outputs into a single cohesive structure.



## 3. Cloud-Rendered Workflow Diagrams (ระบบวาดรูปผังงาน)

- The subagents must analyze the code logic and generate structural flows in Mermaid.js syntax.

- The script must send this Mermaid syntax via an HTTP request to the cloud renderer `https://kroki.io/mermaid/png/` to fetch the diagram as a PNG/SVG image bytes.

- Strictly DO NOT install heavy local diagram tools like Graphviz or mermaid-cli on the host machine.



## 4. Contextual Developer Screenshot Tags (ระบบป้ายเตือนแคปจอ)

- While the subagents are reading the code, if they encounter any visual components, frontend routes, or critical logical triggers, they must automatically insert a highlighted callout box in the Word document.

- The tag must read exactly: `[📸 DEV SCREENSHOT REQUIRED: <Contextual instruction telling the dev exactly what view or state to capture>]`.



## 5. Master Template & Styling Rules

- The script must not create a blank Word file. It must open a predefined `template.docx` file that has pre-styled professional fonts (e.g., Arial or Segoe UI) and custom heading colors, then append the generated content to it.

- If translating code concepts into Thai, always use proper tech terms and avoid awkward literal translations (e.g., use 'API', 'Database', 'Route' directly in Thai text).



# 🔄 Loop Engineering & Quality Control (แนวคิดการวนลูปสร้างโค้ด)

- When generating the files for this codebase (`src/index.js`, `src/parser.js`, etc.), you must follow the Iterative Engineering Loop.

- Automatically execute dry-run tests for every module you write. If any syntax error or npm runtime crash occurs, read the logs, fix the code, and re-test immediately.

- Only stop the generation loop and present the final skill when the entire application executes flawlessly without error.
